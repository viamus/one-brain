# Graph And Correlation

The graph surface shows how memories, entities, explicit links, and inferred similarity relate to
each other. It is designed as a living operator view rather than a static chart.

## Edge Types

| Edge type | Meaning |
| --- | --- |
| `explicit` | A stored memory-to-memory link, such as contains, related, predecessor, or successor. |
| `entity` | Two memories share a named entity extracted during capture or import. |
| `correlation` | Deterministic shared-entity, facet, or vector-neighbor evidence between memories. |

Vector evidence is stored as a `correlation` edge with reason `vector_neighbor`. When vector evidence
combines with shared entities or facets, the edge label can become `semantic_neighbor`. There is no
separate persisted `vector` edge type today.

## Graph Inputs

- Memory metadata: source refs, tags, scope, memory type, confidence, validity windows.
- Entities and relations: explicit facts extracted or submitted with memories.
- Memory links: durable links created by ingestion, grouping, or API calls.
- Embeddings: stored in PostgreSQL with pgvector.
- Aggregates: generated by graph aggregation jobs when grouping opportunities are materialized.

## Deterministic Correlation Pipeline

Correlation scoring and grouping do not use GenAI or an LLM. The graph builder follows a deterministic
pipeline:

1. Select graph memories by query, explicit memory IDs, or scoped filters.
2. Add explicit memory links and entity nodes when requested.
3. Generate shared-entity correlation edges.
4. Add pgvector nearest-neighbor evidence for semantic proximity.
5. Add deterministic facet overlap from titles, metadata, content terms, phrases, tags, and selected
   scope values.
6. Prune by correlation limit, max degree, and score rank.
7. Annotate high-centrality memories as `centroid_candidate` graph roles.
8. Build grouping opportunities around centroid seed memories.
9. Optionally materialize accepted grouping opportunities as aggregate `context` memories.

`centroid_candidate` means a memory has high graph centrality in the current scoped graph. It is not a
generated taxonomy, a durable class, or proof of topic ownership. `centroid_node_id` on a grouping
opportunity is the seed memory used to assemble that candidate cluster.

Memory type classification is separate from graph grouping. Memory types can filter graph inputs, but
grouping opportunities are candidate clusters, not durable classes unless the aggregation job
materializes them as aggregate memories.

## Scores And Versions

Correlation and grouping scores are comparable only within the same scoring algorithm, score version,
scope, and input corpus. Current deterministic scores are versioned as `deterministic-v1` in edge and
grouping metadata.

Before changing weights, thresholds, or formulas:

- keep old score metadata readable;
- bump the score version;
- compare old and new scores in a scoped dry run;
- avoid treating score alone as proof without edge reasons and source evidence.

## Operator Questions

The graph should help answer:

- Which repositories, skills, workflows, or memories are connected?
- Which edges are evidence-backed and which are inferred?
- Which clusters look like business flows?
- Which correlations are noisy and should be filtered?
- Which grouping opportunities deserve materialized aggregate memories?

## Aggregation Job

The graph aggregation job finds grouping opportunities and can materialize them as aggregate
`context` memories.

```powershell
uv run onebrain-jobs aggregate_graph_memories `
  --scope-json '{"catalog":"private-engineering-catalog","source":"github-private-catalog"}' `
  --grouping-limit 25 `
  --correlation-limit 750
```

Inside Docker:

```powershell
docker compose run --rm onebrain-jobs `
  onebrain-jobs aggregate_graph_memories `
  --scope-json '{"catalog":"private-engineering-catalog","source":"github-private-catalog"}' `
  --dry-run `
  --min-score 8 `
  --max-degree 12 `
  --grouping-min-size 3 `
  --grouping-limit 25 `
  --correlation-limit 750
```

Run with `--dry-run` before scheduled materialization. Scoped runs are cheaper and less noisy than
repository-wide runs because the job still performs database reads, facet pair scoring, and pgvector
neighbor searches.

## Reading The Graph

- Start with node labels and memory type.
- Inspect edge reason before trusting a relationship.
- Prefer explicit links and source-ref references over shared generic tags.
- Treat vector-only edges as discovery hints, not final architecture evidence.
- Use graph aggregates as summaries that should preserve source member evidence.
